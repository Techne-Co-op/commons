Pre-Build Audit

Assessment protocol failure — findings and learnings
VERSION 0.1 · JUNE 2026
REGENHUB, LCA · BOULDER, COLORADO
Direction Note · The Self-Describing Commons
Protocol result · Failed

This audit was conducted immediately before W0 substrate work began, after all ten pre-build decisions were recorded. Its purpose was to verify that the repository was self-consistent and ready to build against. It was not. The assessment found three categories of failure — consistency failures, documentation failures, and specification failures — across fifteen specific items. A protocol that runs at the end of a phase and finds fifteen gaps has failed the phase it was designed to protect. The learnings below define what the next phase must do differently.

Scope

Repository: github.com/Techne-Co-op/commons. Published site: techne-co-op.github.io/commons. Audit conducted June 12, 2026.

Remediation status · June 12, 2026

15 of 15 findings remediated. 13 remediated June 12, 2026. CF-02 confirmed resolved June 13 (PyYAML confirmed present in CI; blocker closed). Failure category 1

Consistency failures

The system described itself inaccurately. Published pages showed state that did not match the repository record. The design system documented a standard that the design system page itself did not follow.

CF-01 consistency remediated index.html decision register hardcoded all D1–D10 as “open”

After all ten decisions were recorded in docs/decisions/, the homepage still displayed them as unresolved. A visitor to the published site would conclude the build was blocked by ten open questions. It was not. Remediated: All D1–D10 items updated to “decided” with resolved styling; wave grid chip and intro paragraph updated in the same commit.

Root cause: The decision register is static HTML with no data binding to docs/decisions/ or milestones/index.json. State is maintained in two places and can drift.

Remediated · commit 3230d27 · June 12, 2026 · index.html decision chips updated; all D1–D10 marked decided.
CF-02 consistency remediated STATUS.md shows decision milestones as “pending”; index.json shows “done”

milestones/STATUS.md is generated by scripts/validate.py, which requires PyYAML. PyYAML is unavailable in the current container environment. The file could not be regenerated and fell out of sync when milestone statuses were updated. Status: index.json is authoritative for all automated processes. STATUS.md is a convenience summary and carries no automation weight. It remains stale.

Root cause: A generated artifact with a hard runtime dependency on a library that is not present in the execution environment.

Remediated · June 13, 2026 · PyYAML>=6.0 confirmed present in scripts/requirements.txt; milestone-validate.yml installs it on every CI run. STATUS.md regenerated automatically on push to main. Supabase blocker closed. index.json remains authoritative.
CF-03 consistency remediated design-system.html did not follow the standard it documented

The design system page documented the canonical mode toggle as a text-label button with id="modeBtn" and class .mode-btn. The page’s own button used id="modeToggle", class .mode-toggle, and SVG icons — the older pattern. The page was teaching one thing and doing another. Remediated: Page’s own button updated to the canonical pattern; stale CSS removed.

Root cause: The design system was updated after some pages were written, without a retroactive consistency sweep.

Remediated · commit 3230d27 · June 12, 2026 · design-system.html mode button corrected to canonical pattern.
Failure category 2

Documentation failures

Things were built, configured, or referenced without being documented. A contributor following the written record cannot reconstruct the system from it.

DF-01 documentation remediated docs/SECRETS.md is referenced but does not exist

docs/CONVENTIONS.md references docs/SECRETS.md as the location for secrets handling documentation. The file was never created. An agent or contributor following the onboarding path encounters a dead reference at a critical security boundary.

Remediated · docs/SECRETS.md created · June 12, 2026 · covers secret names, custody, agent protocol, reference-not-value rule.
DF-02 documentation remediated Supabase project is entirely undocumented

The system depends on a live Supabase project (tables, views, Edge Functions, RLS policies). There is no docs/SUPABASE_SETUP.md, no schema DDL in the repository, and no Edge Function source code checked in. The database is a black box. A technical handoff or incident recovery would require reconstructing everything from the dashboard.

For non-technical members: The cooperative’s live database has no written setup instructions. If we needed to rebuild it or hand it to a new technical steward, we would be doing it from memory.

Remediated · docs/SUPABASE_SETUP.md created · June 12, 2026 · both projects documented; ledger_entries schema; v_decision_leverage DDL; edge functions table; CLI setup.
DF-03 documentation remediated C1–C5 contract identifiers are not mapped to files

The Builder’s Collection map references contracts C1 through C11. C6–C11 are clearly mapped to HTML files in the commons site. C1–C5 have no corresponding files and no documentation stating whether they are forthcoming, reserved, or named differently. A contributor reading the map cannot determine whether C1–C5 represent a gap or a deliberate decision.

Remediated · docs/COLLECTION-MAP.md created · June 12, 2026 · explicit C0–C11 to repository files mapping; explains that C1–C5 are existing repo files, not separate HTML artifacts.
DF-04 documentation remediated v_decision_leverage Supabase view is undocumented

loop.html queries the v_decision_leverage view via the Supabase REST API. The view’s DDL, column shape, leverage calculation logic, and RLS policy are not documented anywhere in the repository. If the view is modified or dropped, the loop instrument breaks silently and there is no written definition to restore from.

Remediated · docs/SUPABASE_SETUP.md · June 12, 2026 · v_decision_leverage DDL, column shape, leverage calculation, and RLS policy documented. Marked “inferred” pending confirmation against live dashboard at W0.3.
DF-05 documentation remediated Wave gate milestones directory (milestones/G/) does not exist

The milestone schema includes a G/ directory for gate milestones that govern wave-boundary transitions. The directory does not exist. The wave-boundary authoring guide documents how to write gate files, but none have been written. W0 cannot formally complete without a gate milestone. The gating mechanism is defined but has no records to check.

Remediated · milestones/G/G0-substrate-gate.md created · June 12, 2026 · G0 gate file with all 22 W0 steps in requires_steps and 7 decisions in requires_decisions. G1 gate file also present.
DF-06 documentation remediated C4 synthetic persona fixture pack is absent

The KPI fixture pack (C9) depends on C4 (synthetic persona fixture pack) for realistic member behavior data. C4 does not exist. KPI fixture tests will produce abstract numbers rather than behavior-grounded ones until it does.

Remediated · docs/fixtures/C4-synthetic-personas.md created · June 12, 2026 · five personas (member-a, member-b, admin-1, applicant-1, liaison-1) with fake data, probe usage index, and seed SQL template.
DF-07 documentation remediated Ledger history and audit trail behavior undocumented

The ledger-ingest Edge Function updates the Supabase ledger on every push. Whether it appends (preserving history) or upserts (overwriting) is not documented. For a cooperative accounting system, whether historical states are preserved is a material question. The behavior is unknown from the repository record alone.

For non-technical members: Every time we push a change, our milestone ledger is updated. Whether past states are kept or replaced is an important question for a cooperative that needs auditable records. We do not currently know the answer from our own documentation.

Remediated · docs/SUPABASE_SETUP.md · June 12, 2026 · upsert behavior documented: ledger-ingest upserts on (sha, milestone_id); each push produces a complete snapshot per milestone; historical snapshots preserved. Append semantics confirmed.
Failure category 3

Specification failures

Specifications described things as existing that do not exist, or deferred things without specifying the interim state.

SF-01 specification remediated VERIFICATION.md describes nine CI jobs that have not been implemented

VERIFICATION.md describes a full verification matrix: schema lint, link check, milestone graph validation, HTML validity check, and more. None of these jobs exist in .github/workflows/. The only active workflow is loop-sync.yml. A contributor reading VERIFICATION.md would conclude that automated quality checks are running. They are not. Malformed milestone YAML, invalid status values, and broken internal links can be merged to main without automated detection.

Remediated · docs/VERIFICATION.md updated · June 12, 2026 · status key added (LIVE / PLANNED); milestone-validate and loop-sync marked LIVE with workflow paths; all other jobs marked PLANNED with named milestone.
SF-02 specification remediated D3 decision record omits the canonical repository URL

Decision D3 records that the repository is public from first commit. It does not state the canonical URL. The repository is consistently referenced as github.com/Techne-Co-op/commons across the site, but this is inferred from the GitHub remote, not declared in the decision record.

Already resolved · D3.md contains the canonical URL (github.com/Techne-Co-op/commons). Finding was correct at audit time; D3 was updated before audit publication.
SF-03 specification remediated decision_refs field populated only in W0.1; graph is sparser than intended

The decision_refs field connects decisions to milestones in the dependency graph. It is populated in W0.1 (D3 as a parent) and empty everywhere else. Decisions like D1 (front-end framework, Astro) and D6 (access posture, Supabase auth) are genuine preconditions for multiple W1 milestones but do not appear as graph parents. The build dependency graph is structurally incomplete.

Remediated · June 12, 2026 · decision_refs populated in W0.7 (D10), W0.10 (D5), W0.13 (D1), W1.1 (D6), W2.2 (D9 added alongside existing D2). Five milestone files updated.
SF-04 specification remediated D8 attestation deferral lacks an interim specification

D8 deferred public blockchain attestation to beyond alpha, with a local Merkle tree at the database level noted as a possible interim substitute. The Merkle tree approach is named but not specified or implemented. Until W3 or later, attestation of patronage transactions relies entirely on database trust. The interim mechanism is neither defined nor acknowledged as absent.

Remediated · June 13, 2026 · Decision D11 made: interim attestation via Supabase attest_event() function. Computes SHA-256 of canonical event payload (sorted-key JSON via pgcrypto), stores hash + metadata in attestations table. Tamper-evident audit trail without key custody infrastructure. Upgrade path: sign hash at W3. Migration applied and smoke-tested in CIS project. Thread recorded at commit 0ee283e.
SF-05 specification remediated Loop probe mechanism described conceptually but not specified technically

loop.html presents the probe as a component of the cybernetic loop. Its data source, query shape, update cadence, and role in the CIS architecture are described at a conceptual level only. There is no technical specification. The probe is a named but undefined system component.

Remediated · June 13, 2026 · Decision D12 made: probe spec defined as three assertions against v_decision_leverage — (1) existence: view returns rows, (2) column shape: no null id/decision_status/thread_state, (3) staleness: no thread deliberating >14 days. Implemented in scripts/loop_probe.py (stdlib only, no dependencies) and .github/workflows/loop-probe.yml (runs on push to main + daily 08:03 MT). Verified passing. Thread recorded at commit 0ee283e.
Learnings

Eight learnings for the build phase

These are not retrospective observations. They are forward-looking protocols. Each learning names a behavior change the build phase must adopt to prevent the same category of failure from accumulating.

L-01 learning Documentation is a deliverable, not a follow-up

DF-01 through DF-07 all share one root cause: a thing was built and its documentation was deferred to later. Later never came. Protocol: A milestone is not complete until its documentation is complete. If a milestone creates a Supabase table, the migration DDL and RLS policy are committed in the same PR. If a milestone references a document, that document exists before the milestone closes.

L-02 learning Never describe automation as existing before it does

SF-01 is the clearest failure: VERIFICATION.md reads as though CI jobs are running. They are not. A specification that describes a future state as present is a false record. Protocol: All specification language must be either present-tense-accurate or explicitly marked as “planned” or “W[n] deliverable.” A table in a spec with no “implemented / planned” column is a liability.

L-03 learning Every reference must resolve to a file or an explicit placeholder

DF-01, DF-03, and DF-05 are the same failure: a reference with no referent. Protocol: Before any document or milestone is closed, verify that every reference in it either points to an existing file or is annotated as “reserved / [wave]”. A broken reference at close time is a rejected close.

L-04 learning Static pages that mirror dynamic state require a binding or a flag

CF-01 is a drift problem: a static HTML decision register that cannot update itself when decisions are recorded. Protocol: Any static page that mirrors dynamic state (decision statuses, milestone counts) must either be data-bound (fetch from index.json at load time) or carry a visible “as of [date]” marker so readers know it may be stale. The index.html decision register should be replaced with a live fetch from Supabase in W1.

L-05 learning Design system compliance must be verified before merge, not at audit

CF-03 was found by a manual audit. It should have been caught at the time the page was written or last edited. Protocol: The CI pipeline should include an HTML lint pass that checks for disallowed patterns (e.g., id="modeToggle" appearing in any page that is not the design system itself, duplicate token declarations in <style> blocks that conflict with commons.css).

L-06 learning Deferrals must specify the interim state explicitly

SF-04 (D8 attestation) and SF-05 (loop probe) share a structure: a thing is deferred without defining what holds its place. A deferral that says “we will do X later” without saying “until then, Y is the interim state” creates a gap in the record. Protocol: Every deferral in a decision record must name the interim mechanism, even if the interim mechanism is “none; not yet required.”

L-07 learning Dependency graph completeness is a build input, not a documentation nicety

SF-03 shows a dependency graph that is structurally correct but semantically sparse. The automated gating checks only work if the graph accurately reflects what actually blocks what. An incomplete graph means the gate check will pass milestones that are not genuinely ready. Protocol: Before a wave begins, the decision_refs and depends_on fields for all milestones in that wave must be reviewed and confirmed against the actual work sequence. This is a wave-planning step, not a documentation step.

L-08 learning Generated artifacts must have a regeneration path that works in the execution environment

CF-02 (STATUS.md staleness) was caused by a hard dependency on PyYAML in an environment where Python package installation is not available. Protocol: Before any generated artifact is added to the repository, verify that the generation script can run in CI (the GitHub Actions environment) and in the agent’s execution environment. If it cannot, either port the script or remove the artifact and replace with a CI-generated output.

What held

Five structural elements that survived the audit

The failure framing does not mean the system is in poor shape. The following elements are solid and must be preserved as the build proceeds. The audit found these because they were consistent; that consistency is itself evidence of discipline.

OK-01 solid Milestone schema and validation contract are coherent

The YAML front matter schema, allowed status values, dependency graph fields, and validate.py script are internally consistent. schema.json is the single source of truth and the validator enforces it. This is a strong foundation for the automated CI work ahead.

OK-02 solid loop-sync pipeline is live and verified

Every push to main triggers loop-sync, which POSTs index.json to the Supabase ledger-ingest Edge Function and updates the v_decision_leverage view. Verified across all ten decision commits. The cybernetic loop between the repository and the live instrument is functioning.

OK-03 solid Design system is consistent across all fourteen pages (post-remediation)

All fourteen HTML pages now share commons.css, use the unified commons-mode localStorage key for dark/light preference, and follow one of three documented layout grammars (index, instrument, document). The one self-inconsistency identified by this audit has been remediated.

OK-04 solid Decision record format is clear and human-auditable

The docs/decisions/ files have a consistent structure: front matter with status and decided_by, context, proposed default, and record of decision. All ten pre-build decisions follow this format. A non-technical member can read any file and understand who decided what, when, and why.

OK-05 solid Agent operating contract is specific and enforced

AGENTS.md covers session start sequence, file edit protocol, commit signing, status vocabulary, agent boundaries, and Workshop coordination. It is detailed enough that an agent can operate without ambiguity. The “agents never set decision status” boundary is explicit and was consistently observed across all ten decision commits.

Appendix

Glossary for non-technical members

Repository — The shared project folder (on GitHub) where all files live: milestone records, decision logs, site pages, scripts, and automation rules. Every change is recorded with a timestamp and author.

Milestone — A specific, verifiable piece of work with a status (pending, in progress, done), a list of dependencies, and a definition of what “done” means.

Wave — A phase of the build. W0 is the substrate (foundation). W1–W6 are successive layers. Each wave gates the next.

CI (Continuous Integration) — Automated quality checks that run whenever anyone proposes a change. These prevent errors from entering the shared record without a human having to catch them manually.

Supabase — The cloud database behind the live instruments. It stores the ledger and serves it to the published site.

DDL (Data Definition Language) — The instructions that define the shape of a database: which tables exist, what columns they have, and who can read or write them. Committing DDL to the repository means the database can be rebuilt from the record.

RLS (Row-Level Security) — The database permission layer. It controls which users can see which rows. This is how public KPIs are made visible while member data stays private.

Gate milestone — A checkpoint milestone at the end of a wave. It verifies that all wave work is complete before the next wave can begin. Like a quality inspection before the next production run.
