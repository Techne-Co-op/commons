Pre-Build Audit

Self-referential consistency, documentation & architectural gaps
VERSION 0.1 · JUNE 2026
REGENHUB, LCA · BOULDER, COLORADO

Completed before the W0 substrate build begins. This document surveys what exists, what claims what, and where the gaps are. Written for both technical contributors and non-technical co-op members.

Scope

Repository: github.com/Techne-Co-op/commons. Published site: techne-co-op.github.io/commons. Audit conducted June 2026 immediately after all 10 pre-build decisions (D1–D10) were recorded.

The audit asks three questions: (1) Does the system describe itself accurately? (2) What is missing that should be documented? (3) What structural pieces are absent that the build phase requires?

Section 1

Self-referential consistency

These are places where the system’s published descriptions did not match its actual state at the time of the audit. All three have been remediated in the same session.

C-01 resolved index.html decision register showed all D1–D10 as “open”

The homepage’s decision register hardcoded the word “open” for all ten pre-build decisions. After all ten decisions were formally recorded in docs/decisions/ in June 2026, the page became stale. Fix: Updated all D1–D10 items to “decided” with the .resolved CSS class (green). The wave grid chip was updated from “10 open” to “10 decided”. The accompanying paragraph was rewritten to reflect the current state.

For non-technical members: The front page of the commons site was still showing our ten planning decisions as unresolved. It now reflects that all ten have been answered.

C-02 resolved STATUS.md decision entries show “pending” while index.json shows “done”

milestones/STATUS.md is generated by scripts/validate.py, which requires PyYAML. PyYAML is not available in the current container environment (no pip access). As a result STATUS.md is out of sync with milestones/index.json, which correctly shows all ten decisions as done. Status: index.json is authoritative (it’s what the loop-sync GitHub Action reads to update Supabase). STATUS.md is a human-readable summary only and carries no automation weight. A future session with Python available, or a Node.js port of validate.py, should regenerate it.

For non-technical members: There’s a summary table in the repository that hasn’t been refreshed. It doesn’t affect any live systems, but it should be updated when the tooling is available.

C-03 resolved design-system.html was self-inconsistent between its own button and its canonical code example

The design system page documents the canonical mode toggle as a text-label button (id="modeBtn"). Its own mode toggle was using the older SVG+label pattern (id="modeToggle") with a different element ID. Fix: The page’s own button was updated to match the standard it documents. Stale .mode-toggle CSS was removed.

For non-technical members: The page that defines how our interface should look was not following its own rules. That has been corrected.

Section 2

Documentation gaps

These are pieces of documentation that are referenced or implied by existing files but do not yet exist. They are not necessarily blocking, but each represents an assumption that a contributor or agent would have to resolve by inference.

D-01 gap docs/SECRETS.md is referenced but does not exist

docs/CONVENTIONS.md references docs/SECRETS.md as the location for secrets handling documentation. The file does not exist in the repository. Any contributor or agent following the conventions file encounters a dead link. Recommendation: Create docs/SECRETS.md documenting the reference-not-value protocol: secrets live in .secrets/, are read at runtime via cat, and are never written into workspace files or commits.

D-02 gap Supabase setup procedure is entirely undocumented

The system depends on a Supabase project (project ID: hvbdpgkdcdskhpbdeeim) with specific tables, views, RLS policies, and Edge Functions. There is no docs/SUPABASE_SETUP.md or equivalent. A new contributor has no path to recreate or audit the database schema without direct access to the Supabase dashboard. The v_decision_leverage view queried by loop.html, the ledger-ingest Edge Function, and the agent_direct_messages table are all undocumented. Recommendation: Create docs/SUPABASE_SETUP.md with table definitions, view DDL, and Edge Function entry points. Dump current schema DDL and commit it to supabase/migrations/.

For non-technical members: We rely on a cloud database that has no written setup instructions. If we needed to rebuild or hand off to a new technical steward, we’d be reconstructing it from memory.

D-03 gap C1–C5 contract identifiers are not mapped to actual files

The milestone system references contracts C1 through C11. C6–C11 are clearly mapped to HTML files in the commons site (e.g., c6-schema-reference-card-v0.1.html). C1–C5 are not present as files or documented anywhere in the repository. It is unclear whether they represent documents that have not yet been created, documents with different naming, or a numbering scheme that starts at C6 for this phase. Recommendation: Add a mapping section to docs/CONVENTIONS.md or a new docs/CONTRACTS.md listing C1–C5 with their intended document names or explicit “reserved / future wave” status.

D-04 gap VERIFICATION.md lists CI jobs that do not exist

VERIFICATION.md describes a verification matrix with nine CI job types (schema lint, link check, milestone graph validation, etc.). None of these jobs exist in .github/workflows/. The only active workflow is loop-sync.yml, which pushes milestones/index.json to Supabase on every push to main. Recommendation: Either implement the CI jobs in W0 (the schema lint is the highest-value first candidate), or update VERIFICATION.md to distinguish “planned” from “implemented” jobs so contributors aren’t confused about what is actually running.

For non-technical members: We have a document describing automated quality checks, but the checks themselves haven’t been built yet. This is normal for an early-stage system, but the document implies they exist.

D-05 gap v_decision_leverage Supabase view is undocumented

loop.html queries v_decision_leverage via the Supabase REST API to show open decisions ranked by leverage. The view’s DDL, what columns it exposes, how it calculates leverage, and what RLS policy governs it are not documented anywhere in the repository. If the view is dropped or modified, the loop instrument breaks silently. Recommendation: Document the view definition in docs/SUPABASE_SETUP.md and add a note in the loop instrument source that names the view and its expected shape.

D-06 gap Wave-boundary briefs not authored (milestones/G/ missing)

The milestone schema includes a G/ subdirectory for gate milestones that govern wave-boundary transitions. This directory does not exist. The wave-boundary authoring guide (c10-wave-boundary-authoring-guide-v0.1.html) documents how to write these, but none have been written. W0 concludes with a gate milestone (W0.G) that verifies substrate readiness before W1 begins. Without the gate file, the gating mechanism has no record to check. Recommendation: Author milestones/G/W0.G-substrate-gate.md as part of W0 setup, following the authoring guide.

D-07 gap C4 synthetic persona fixture pack is absent

The milestone C9-kpi-fixture-pack-v0.1.html references C4 (the synthetic persona fixture pack) as a dependency for realistic KPI testing. C4 does not exist as a file. Without it, KPI fixture tests will use abstract numbers rather than realistic member behavior patterns. Recommendation: Create the C4 synthetic persona fixture pack in W0 or early W1, covering at least three distinct member archetypes (e.g., high-patronage individual, low-activity individual, organizational member).

Section 3

Architectural gaps

These are structural pieces that the system will need to function as described but that have not yet been built or specified in sufficient detail. Most are W0 deliverables; they are named here so the W0 work plan can address them deliberately.

A-01 noted CI pipeline does not exist beyond loop-sync

The only GitHub Actions workflow is loop-sync.yml. There is no schema lint, no broken-link check, no milestone graph validation, and no HTML validity check. These are all described in VERIFICATION.md as automated. The practical risk is that malformed milestone YAML, invalid status values, or broken internal links can be merged to main without any automated catch. W0 deliverable: At minimum, a schema lint job that runs scripts/validate.py (or a Node.js equivalent) on every PR and push to main.

A-02 noted D3 record does not specify the canonical repository URL

Decision D3 (repository visibility) records that the repository is public from first commit, but does not name the canonical URL. The repository is currently at github.com/Techne-Co-op/commons; this is consistent across the site but is implicit, not declared. Recommendation: Add one line to docs/decisions/D3.md specifying the canonical URL so it is on the record and not inferred from the GitHub remote.

A-03 noted decision_refs field is described but unpopulated in most milestones

The milestone schema includes a decision_refs array field that adds decisions to a milestone’s dependency graph. It is populated in W0.1 (which has D3 as a parent via this field) but is empty in all other milestones. This means the dependency graph is sparser than intended. Decisions like D1 (front-end framework) should arguably appear as parents of W1 milestones that depend on Astro. Recommendation: As part of W0, do a pass through the milestone graph and populate decision_refs where a decision is a genuine precondition for a milestone’s work.

A-04 noted No agent heartbeat or presence mechanism exists

AGENTS.md describes a Workshop coordination protocol including presence heartbeats. The Workshop API is live, but there is no scheduled task or cron job in the commons repository that sends heartbeats. Nou’s Workshop state lives in the agent’s runtime session files and is not durably recorded in the repository. Recommendation: The heartbeat mechanism is out of scope for the commons repository itself (it belongs to the agent configuration), but the commons AGENTS.md should clarify that heartbeats are an agent-session responsibility, not a repository CI responsibility.

For non-technical members: Our agents are supposed to check in regularly to a shared workspace. That check-in system is configured in the agents’ runtime environment, not in this repository. The documentation should say so clearly.

A-05 noted No export or archive mechanism for milestones/index.json history

The Supabase ledger is updated on every push via loop-sync. The ledger-ingest Edge Function presumably upserts records. Whether the previous state is preserved (an append-only audit trail) or overwritten (last-write-wins) is not documented. For a cooperative accounting system, auditability of state changes is significant. Recommendation: Document the ingest function’s upsert behavior. If it overwrites, add a ledger_history table or use Supabase’s point-in-time recovery as the audit mechanism.

For non-technical members: Our milestone ledger gets updated every time we push changes to the repository. Whether it keeps a history of past states, or just the current one, is an important question for a cooperative that needs to be able to audit its own records.

A-06 noted Attestation mechanism (D8) deferred; no interim substitute specified

D8 deferred public blockchain attestation to beyond alpha, with a local Merkle tree at the database level as an interim substitute. The Merkle tree approach is not specified or implemented. Until W3 or later, attestation of patronage transactions is entirely trust-based (recorded in the database, no cryptographic commitment). This is acceptable for alpha, but the decision record should be updated to name the interim mechanism explicitly (e.g., “SHA-256 content hash stored in attestation_hash column of the transactions table”). Recommendation: Before W3, define the interim Merkle approach in enough detail that it can be implemented in W2.3.

A-07 noted loop.html probe mechanism unspecified

loop.html shows a cybernetic loop diagram with a “probe” element. The probe’s data source, query shape, and update cadence are not specified anywhere in the repository or associated contract files. The loop instrument currently renders the leverage view from Supabase but the probe’s meaning in the CIS architecture is only described at a high conceptual level. Recommendation: Add a technical specification for the loop probe to c8-nou-gateway-spec-v0.1.html or a new dedicated contract, defining what data the probe surfaces and how often it updates.

A-08 noted Astro build pipeline is decided but not scaffolded

D1 decided on Astro as the front-end framework. The milestones/W1/ directory contains W1 milestones describing the Astro scaffolding work, but no Astro project exists in the repository. The current site is static HTML. The W0 substrate work includes setting up the build toolchain; this gap is expected and on-roadmap. It is noted here for completeness. W0 deliverable: W0.4 covers Astro scaffolding and is the appropriate place for this work.

For non-technical members: We’ve decided which tool we’ll use to build the member-facing site. The current pages are hand-written. The W0 phase will set up the proper build process.

Section 4

Build readiness assessment

W0 substrate can begin. The pre-build decision phase is complete. The gaps identified above are either W0 deliverables (and thus expected) or documentation work that can proceed in parallel without blocking build steps.

The highest-priority items before W1 can begin:

P-01 before W1 Create docs/SECRETS.md (D-01)

Referenced but absent. Any agent or contributor onboarding follows CONVENTIONS.md and hits a dead reference. Low effort, high orientation value.

P-02 before W1 Document Supabase schema and Edge Functions (D-02, D-05)

Without this, the database is a black box. Any incident or handoff is significantly harder. W0.2 (Supabase project setup) is the natural home for this documentation.

P-03 before W1 Implement schema lint CI job (A-01, D-04)

The validate.py script already exists but cannot run in CI without Python. Either port the validation to Node.js (no additional dependency) or add a Python setup step to the GitHub Action. This is the single highest-leverage CI investment: it catches malformed milestone YAML before it reaches main.

P-04 before W1 Author W0.G gate milestone (D-06)

The wave-gate mechanism has no records to check until the gate files exist. W0 cannot formally complete without a gate milestone to verify against.

Section 5

What is working well

The audit is not only a list of gaps. The following structural elements are solid and should be preserved as the build proceeds.

W-01 solid Milestone schema and validation contract are coherent

The YAML front matter schema, the allowed status values, the dependency graph fields, and the validate.py script are internally consistent. The schema.json is the single source of truth and the validator enforces it. This is a strong foundation for the automated CI work ahead.

W-02 solid loop-sync pipeline is live and tested

Every push to main triggers loop-sync, which POSTs index.json to the Supabase ledger-ingest Edge Function and updates the v_decision_leverage view. This was verified across all ten decision commits. The cybernetic loop between the repository and the live loop instrument is functioning.

W-03 solid Design system is internally consistent across all pages

All 14 HTML pages now share commons.css, use the unified commons-mode localStorage key for dark/light preference, and follow one of three documented layout grammars (index, instrument, document). The design-system.html self-inconsistency was the only remaining gap and has been resolved.

W-04 solid Decision record format is clear and human-auditable

The docs/decisions/ files have a consistent structure: front matter with status and decided_by, a context section, a proposed default, and a record of decision. All ten pre-build decisions follow this format. Non-technical members can read any decision file and understand who decided what, when, and why.

W-05 solid Agent operating contract is specific and comprehensive

AGENTS.md covers session start sequence, file edit protocol, commit signing, status vocabulary, what agents may and may not do, and the Workshop coordination protocol. It is detailed enough that a new agent can operate without ambiguity about its role. The “agents never set [decision] status” boundary is explicit and consistently enforced.

Appendix

Glossary for non-technical members

Repository — The shared folder (on GitHub) where all project files live: milestone records, decision logs, site pages, scripts, and automation rules. Think of it as the cooperative’s filing cabinet, but one where every change is recorded with a timestamp and author.

Milestone — A specific, verifiable piece of work. Each milestone has a status (pending, in progress, done), a list of what it depends on, and a definition of what “done” means. The 70 milestones are organized into waves.

Wave — A phase of the build. W0 is the substrate (foundation). W1–W6 are successive layers. Each wave gates the next: W0 must be complete and verified before W1 begins.

Decision (D1–D10) — A question that needed an answer before building could begin. Who holds the lease? Which tools do we use? Are member data public or private? All ten have been answered and recorded by the organizers.

Supabase — The cloud database behind the live instruments (the HUD, the loop, the KPI fixtures). It stores the ledger data and serves it to the published site.

loop-sync — An automated rule that runs every time a change is saved to the repository. It reads the milestone ledger and sends the updated state to the database, so the live instruments stay current.

CI (Continuous Integration) — Automated quality checks that run whenever anyone proposes a change. These are partly built (loop-sync) and partly planned (schema lint, link checks). Having CI catch errors automatically means human reviewers spend their time on judgment calls, not mechanical checks.

RLS (Row-Level Security) — The database’s permission layer. It controls which users can see which rows. For the cooperative, this is how we make some data public (anonymized KPIs) while keeping member data private unless you’re logged in.
