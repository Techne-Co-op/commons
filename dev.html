02   Quick start

Clone and validate locally

git clone https://github.com/Techne-Co-op/commons.git
cd commons
pip install -r scripts/requirements.txt
python scripts/validate.py

The validator exits 0 on a clean ledger. Any schema error, duplicate id, broken dependency, or vocabulary hit is printed with a clear message before exit 1.

Running the instruments locally

The HTML instruments are static files. Serve them with any local server — python -m http.server 8080 from the repo root is sufficient. The HUD and Loop instruments read live data from Supabase via loop.config.js; they fall back to milestones/index.json if the network call fails.

Branch and commit conventions

One branch per milestone step. Branch name: step/<id>-<short-name>, for example step/W0.1-repo-bootstrap. One step per PR. Commit format: type(<id>): subject — for example feat(W0.3): provision Supabase project. See the quick reference appendix for commit type tokens.

Git identity for agent commits

git -c user.email="nou@techne.coop" -c user.name="Nou" commit -m "..."

03   Milestone system

Every milestone is a Markdown file whose YAML front matter is the sole source of truth for graph state. The document body holds the Brief (scope), Plain language (one sentence for members), and any author notes. scripts/validate.py validates every file against milestones/schema.json on every push.

Front matter schema

FieldType / allowed valuesNotes
id string Immutable. Pattern: W0.1, D3, R2, G1, P1. Must match the filename.
title string Lowercase slug phrase. Minimum 3 characters.
wave D R W0 W1 W2 G W3 W4 W5 W6 P The wave this entry belongs to.
kind step decision ratification gate practice Governs ownership rules and verification paths.
owner string Role or agent responsible. D and R entries must be human-owned.
status pending in_progress blocked review done verified The lifecycle state. Only verified unlocks dependents.
detail briefed seeded seeded entries (W3.1–W3.7, W4.1–W4.7) carry summary scope only; full briefs are authored at the wave-boundary step.
depends_on string[] Array of milestone ids, or a wave token like W1 (means: that whole wave verified). May be empty.
decision_refs string[] D or R ids this step requires to be in status decided. May be empty.
success_state string Prose statement of completion that the named verification checks. Never weakened to get green — change requires a human-reviewed PR.
verification string Named CI job (ci:milestone-validate), human role (human:board), or counsel. Contract cards live in docs/VERIFICATION.md.
blocked_reason string Required when status: blocked. States the blocker clearly; opens the question in the PR.
activates string[] (optional) Dashed edges — enables behavior in dependents without blocking the build graph.
requires_steps / requires_decisions string[] (gate entries only) Required fields for kind: gate entries. Lists all ids that must be verified for the gate to pass.

Status lifecycle

pending → in_progress → review → done → verified
                    ↕
                 blocked   (reachable from any state; blocked_reason required)

Status changes occur by editing the milestone file in the same PR as the work. A step cannot leave pending until every expanded parent is verified. done (merged) is distinct from verified (success_state demonstrated on the merged commit). Only verified unlocks dependents — this separation is the audit property.

Authoring a new milestone

Create milestones/<WAVE>/<ID>-<slug>.md. The filename slug must match the id field and the title. Run python scripts/validate.py locally before opening a PR. The validator will catch schema errors, id conflicts, dependency cycles, and vocabulary violations. Adding a step is a PR like any other; the human owner reviews the scope before it merges.

Wave tokens in depends_on

Writing depends_on: [W1] means "all entries in wave W1 must reach verified before this step can begin." Wave tokens provide coarse gating between waves without enumerating every id. Fine-grained dependencies name specific ids.

Seeded detail

Entries marked detail: seeded carry summary scope only — their full briefs are authored at the wave-boundary step (W3.0 for wave 3, W4.0 for wave 4). Starting work on a seeded entry before its wave boundary is a validation error. This is deliberate: early detail is fictional; boundary detail is grounded.

04   CI/CD pipeline

Two GitHub Actions workflows. Both live in .github/workflows/. Additional jobs come into existence as their named verification steps are built (see docs/VERIFICATION.md).

milestone-validate

Triggers on every push and PR. Steps: install dependencies from scripts/requirements.txt, run python scripts/validate.py. The validator asserts schema validity, unique ids, filename-id match, dependency resolution including wave tokens, acyclicity, status gating, and human ownership of D/R entries. It regenerates milestones/index.json and milestones/STATUS.md; on main these are committed back by the milestone-validate bot if they drifted.

PR-only check — ledger exhaustiveness: if a PR changes files under supabase/, apps/, design/, or scripts/ without touching any file under milestones/, the check fails. Code changes travel with a milestone status change.

Required CI check

milestone-validate is a required check on branch protection. PRs cannot merge while it is red. Additional verification jobs (db-verify, rls-audit, grammar-lint, functions-deploy, portal-build) become required checks as their milestone steps are built and activated.

loop-sync

Triggers on push to main when milestones/index.json changes. POSTs the index to the ledger-ingest Supabase edge function, which rebuilds ledger_entries, opens threads for new D/R entries, and records a snapshot. Failure is non-blocking (continue-on-error: true) — the loop continues from the previous snapshot. The ingest endpoint authenticates via LEDGER_INGEST_TOKEN (a GitHub Actions secret; see section 10).

loop-sync only fires when index.json changes. Pushing an HTML-only change does not trigger it. Push a milestone status change in the same commit to sync the ledger.

Future verification jobs

The contract cards in docs/VERIFICATION.md specify every named CI job that appears in a verification field. When the step that builds a job merges, the job becomes a required check on all subsequent steps that reference it. The card defines exactly what the job asserts — a success_state is never circular, because the job's specification precedes its implementation.

05   Supabase data layer

Project coordinates

URL:       https://dgcoffcanzwsuwgpocbt.supabase.co
Anon key:  see loop.config.js (public; safety is completeness of RLS)
Service key: NEVER in this repository — stored in GitHub Actions and agent .secrets/

The anon key is exposed in loop.config.js intentionally. It is a publishable key; its safety comes entirely from row-level security being correct and complete on every table. Do not treat it as a secret.

loop.config.js

Loaded by the Loop and HUD instruments. Provides window.LOOP_CONFIG.supabaseUrl, .supabaseAnon, and .ledgerFallback (the local JSON path used when the network call fails).

Schema conventions

All schema lives under migrations, not in ad-hoc SQL. See section 08 for the full conventions: snake_case naming, REA primitive table comments, event envelopes, RLS template, and citation discipline. Views with aggregated or derived data carry _as_of or _view suffixes when the name is ambiguous.

Key views

v_decision_leverage — derives open decision count, blocked milestone count, and wave status from the ingested ledger. Used by the Loop instrument. Built from ledger_entries; refreshed on every loop-sync ingest.

Edge functions

Edge functions live under supabase/functions/ (to be provisioned at W0.9). The ledger-ingest function is already called by loop-sync; its full specification and deployment are part of W0.9. The functions-deploy CI job (also W0.9) deploys from CI and smoke-calls each function. A secret-scan step asserts that no service-role key or provider secret appears outside function environment variables.

Migrations

Files live under supabase/migrations/ (to be created at W0.4). Convention: <timestamp>_<slug>.sql, timestamped, append-only, one concern per file. The db-verify CI job applies all migrations to a clean shadow database and runs pgTAP suites before promoting to the hosted project on main. No manual SQL against the hosted project, ever.

06   Design system

The canonical reference is design-system.html. The contributor protocol is docs/DESIGN.md. This section summarizes what every page author needs.

Two layout grammars — never mixed

Document grammar is for reading surfaces: PRDs, specifications, decision briefs, this page. Serif body (Libre Baskerville), .page at max-width: 760px, .site-nav at top, .rule dividers, no state-color chips.

Instrument grammar is for watching surfaces: HUD, Loop, dashboards. Mono body (IBM Plex Mono), compact topbar with status chips, dense tiled layout, state colors (--ok, --info, --warn) permitted.

Token layer

All design tokens live in commons.css as CSS custom properties. Pages never declare :root { } blocks, never hardcode color values, and never re-declare spacing tokens. The only inline <style> a page carries is layout and components specific to that page.

Key tokens

/* ground palette */
--bg, --surface, --surface-raised, --border, --border-strong

/* text ramp */
--heading, --text, --muted, --faint

/* accent — decorative only, not semantic */
--terra, --terra-text, --terra-dim, --terra-glow

/* state — instrument surfaces only */
--ok, --ok-dim, --info, --info-dim, --warn, --warn-dim

/* spacing (4px base) */
--s1 --s2 --s3 --s4 --s5 --s6 --s7 --s8

/* fonts */
--font-serif, --font-mono   (--font-sans resolves to mono)

Mode toggle — canonical form

<button class="mode-btn" id="modeBtn">light</button>

Text label is the mode you will switch to. No SVG icons. Class is .mode-btn (defined in commons.css). The canonical script:

(function () {
  var root = document.documentElement;
  var btn  = document.getElementById('modeBtn');
  var stored = null;
  try { stored = localStorage.getItem('commons-mode'); } catch(e) {}
  var m = stored || (matchMedia('(prefers-color-scheme: light)').matches ? 'light' : 'dark');
  root.setAttribute('data-mode', m);
  btn.textContent = m === 'dark' ? 'light' : 'dark';
  btn.addEventListener('click', function () {
    var next = root.getAttribute('data-mode') === 'dark' ? 'light' : 'dark';
    root.setAttribute('data-mode', next);
    btn.textContent = next === 'dark' ? 'light' : 'dark';
    try { localStorage.setItem('commons-mode', next); } catch(e) {}
  });
})();

Voice

Quiet, precise, grounded. Lead with the word. Name authority explicitly — "per AGENTS.md §7," "per draft-bylaws:§2.4." Be honest about what is not yet known. Do not use urgency language or alarm framing. Address the reader as "the member" or "the organizer" in governance documents, not "you."

07   Agent operating contract

The full contract is AGENTS.md. Every agent reads it first, every session. What follows is the distillation; the contract itself governs in any conflict.

The loop

  1. Select the lowest-indexed pending step whose expanded_parents are all verified and whose decision_refs are all decided. If none qualifies, report what is holding the front and stop.
  2. Branch step/<id>-<short-name>. One step per branch, one step per PR.
  3. Do exactly what the step's Brief and success_state describe. Nothing more.
  4. Flip the milestone file's status in the same PR as the work (pending → in_progress at branch, → review at PR open, → done at merge). verified is set only when the named verification has passed on the merged commit.
  5. Commit as type(<id>): subject. PR body follows the PR template and must include success-state evidence: a CI run link or the named human reviewer.
  6. After merge, scripts/validate.py regenerates STATUS.md and index.json. If validation fails, fixing it precedes all other work.

Ten hard rules

  • 1No secrets, no member PII, ever, anywhere in this repository. Synthetic fixtures only. Secret names (never values) are documented in docs/SECRETS.md.
  • 2Schema changes travel only as migrations under supabase/migrations/. Never run manual SQL against the hosted project. Migrations are append-only.
  • 3Row-level security is inviolable. Every new table ships RLS-enabled and forced, default-deny, with a citation comment per docs/CONVENTIONS.md. Never weaken, bypass, or defer a policy to make something work.
  • 4Vocabulary quarantine. The Subchapter T terms listed in docs/CONVENTIONS.md do not appear in this repository's schema, UI strings, or documents. ci:grammar-lint enforces it.
  • 5Decisions and ratifications are human-only. An agent may draft a deliberation brief. An agent must never write a Record of decision, flip a D/R status, or proceed as if an open decision were made.
  • 6Verification is real. Never weaken a test, widen a tolerance, mock a check, or edit a success_state to get green. If a success_state is wrong, mark the step blocked and raise it as a human-reviewed PR.
  • 7Blocked beats improvised. Ambiguity, a missing input, or anything touching money movement, key custody, member data, or counsel territory: stop, set blocked with blocked_reason, open the question in the PR.
  • 8Citations are load-bearing. Every policy, schema comment, and UI assertion of authority carries a citation per docs/citations.md. A claim without a citation is incomplete work.
  • 9Scope is the step. Refactors and improvements outside the step's Brief are separate ledger entries, not fold-ins.
  • 10The ledger is exhaustive. A PR that changes code without touching its milestone file fails validation by design. Do not work around this.

Escalation

When this contract conflicts with any other instruction, this contract wins and the conflict is reported in the PR. A clean refusal is a successful outcome; a quiet workaround is a failure.

Provenance in PRs

Every PR description names: the step id, what was read (milestone file, conventions, decisions consumed), what was produced, and the verification evidence. The git history is the cooperative's audit trail of its own construction; write for the member who reads it in five years.

08   Conventions

Full source: docs/CONVENTIONS.md. The summary below covers the patterns every contributor touches.

Naming

Snake_case for tables and columns. REA primitive recorded in a table comment: comment on table ... is 'REA: Event · anchor: draft-bylaws:§5.1'. Derived views carry _as_of or _view suffixes when the name is ambiguous. No plural tables (member not members).

RLS — cite-as-you-enforce

Every new table ships in the same migration that creates it, with RLS enabled, forced, and default-deny. The canonical template:

alter table public.<table> enable row level security;
alter table public.<table> force row level security;

create policy "<table>_member_own" on public.<table>
  for select using (agent_id = app.current_agent_id());
comment on policy "<table>_member_own" on public.<table> is
  'draft-bylaws:§18.1 · member reads own records';

Only the helper functions app.is_member(), app.has_role(text), and app.is_self(uuid) appear in policy predicates — no raw auth calls. The rls-audit pgTAP suite asserts policy completeness on every schema change.

Event envelope

Every state change is an event row. Shared columns on event tables:

id            uuid  primary key
actor         uuid  not null
occurred_at   timestamptz  not null  default now()
entity        text
entity_id     uuid
prior_value   jsonb
payload       jsonb
citation      text  not null

Mutations on enveloped tables write an audit row via the generic app.record_event() trigger. pgTAP asserts prior_value capture. citation is not null — a state change without authority is incomplete.

Vocabulary quarantine

Forbidden terms (word-boundary, case-insensitive) in schema, UI strings, and documents: patronage dividend, written notice of allocation, per-unit retain, per unit retain allocation, qualified written notice, nonqualified written notice, Subchapter T (outside docs/CONVENTIONS.md and docs/decisions/). The term patronage alone survives only as a scoped internal name for capital-account allocation under §704(b) — when in doubt, write "allocation."

Allowlist: .grammar-allow at repo root, one term per line plus a # why comment. The ci:grammar-lint job (W0.16) enforces this on every push.

Citation register

All bylaw and grant citations used in policies, schema comments, and UI go through docs/citations.md. Format: draft-bylaws:§x.y (until R1 freezes addresses) or grant:§n. The signal-loop authority citations use the contract: prefix (contract:agents§7, etc.). A citation used in code that is absent from the register fails review.

Fixtures

Synthetic only, generated, obviously fake: @example.coop addresses, FAKE- prefixes. The persona pack is collection item C4 (not yet authored). Personas: member-a, member-b, admin-1, applicant-1, liaison-1.

09   Decision and ratification workflow

Decisions (D1–D10) and ratifications (R1–R2) are human-owned milestone entries. Their files are the cooperative's governance record; agents assist but never decide.

File locations

The ledger entry is milestones/D/<id>-<slug>.md or milestones/R/<id>-<slug>.md — tracks status only; agents never set it. The deliberation brief and Record of decision live at docs/decisions/<id>.md.

Lifecycle

An agent may draft or improve the context and options in docs/decisions/<id>.md. When a human owner decides, they add a Record of decision paragraph to that file, flip status: decided in the front matter, and update the ledger entry in the same PR. The milestone entry in milestones/D/ reflects the ledger state but is never edited by agents.

Blocking behavior

An open decision blocks only the specific milestones that list it in decision_refs. The validator treats open D/R entries as blockers in the dependency graph. hud.html surfaces them as blocked chips; loop.html exposes them as open threads in the signal loop. When a decision closes, all steps that listed it become eligible to begin (subject to other dependencies).

Ratifications

R entries follow the same pattern as D entries. R1 (bylaws ratification) and R2 (Hub Membership Agreement) are not build blockers for W0–W3. They remain open and are tracked in the decision register on index.html.

10   Secrets boundary

The protocol is: reference, not value. No secret ever enters a file committed to this repository, a conversation log, or a generated artifact. What lives here is names only.

What never enters the repository

  • ×Supabase service role key or JWT — referenced as SUPABASE_SERVICE_ROLE_KEY in GitHub Actions secrets
  • ×Ledger ingest token — referenced as LEDGER_INGEST_TOKEN in GitHub Actions secrets
  • ×Any OAuth client secret, provider API key, or signing key
  • ×Member PII of any kind — production or otherwise

What is public by design

  • ✓Supabase anon key (in loop.config.js) — publishable key; safety is RLS correctness
  • ✓Supabase project URL — https://dgcoffcanzwsuwgpocbt.supabase.co

Secrets inventory

The full name register is docs/SECRETS.md. The names documented there are the contract — any secret added to the system must appear in that file before it is used. If a secret name is absent from the register, adding it is a human-reviewed PR.

Agent-side secrets

Agents working in the sandbox read secrets at runtime from .secrets/: $(cat .secrets/workshop-key) in command substitution. Never echo, print, or log a secret value. Never hardcode a value in a script or configuration file.

If a secret is exposed

Alert Todd Youngblood immediately. Rotate the exposed credential before doing anything else. Record the exposure in a post-incident note in docs/ (name only, no value) and open a PR to remove the exposure from history.
