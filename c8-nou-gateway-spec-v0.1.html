Nou Gateway Spec

The three-constraint harness as implementable detail

Nou v1 is a path, not a person: one Edge Function through which a member's own question travels to a language model and back, seeing exactly what the member could see, citing everything it says, and leaving a record of having been asked. This document specifies that path closely enough for an agent to build it.

01 · Frame

The PRD commits three constraints (CIS §11, §07): Nou inherits the permissions of the invoking user, never more, with all actions logged; answers are grounded with citations to section and record; and where grounding fails, Nou refuses rather than improvises. Phase 01 scope is read-only member-facing Q&A over governing documents and the member's own records. Everything else here, the wire shapes, the verification pass, the wording, is proposal in service of those commitments, tagged prop and adopted the way matrix cells are: by PR at W1.7.

A worthwhile honesty note, since this document is Nou specifying Nou: nothing below grants the gateway judgment. The harness is designed so that the interesting properties are structural, enforced by the database and the verification pass, not promised by a prompt. Where a property depends on the model behaving well, this spec says so out loud.

02 · Shape of a Call

One function, nou-gateway, deployed from supabase/functions under the W0.9 scaffold, JWT verification on, CORS shared. The call passes through six stages, and the client identity in play at each stage is the whole security story.

stageacts aswhat happens
1 verify—JWT verified by the W0.9 middleware; role claims read; anonymous calls rejected plan
2 retrieveuser-scoped clienta Supabase client constructed with the caller's JWT fetches governing documents and, if asked, the caller's own records; RLS does the scoping plan
3 generate—retrieved context plus the question go to the Claude API; ANTHROPIC_API_KEY exists only in function env per SECRETS.md plan
4 verify citationsuser-scoped clientevery citation the model emits is resolved through the same user client; what the caller cannot read is stripped prop
5 loglog writerexactly one row enters nou_interactions: prompt, citation set, response, grounding outcome plan
6 return—the response shape of §03, including the interaction id so the member can point at the log entry prop

The dual-client pattern in stages 2 through 5 is the spec's central mechanism. All reading happens through the user-scoped client, with no exceptions and no fallback; the privileged client appears exactly once, wrapped in a single-purpose log_interaction() helper that exposes insert and nothing else. Finding F10 names the consequence: the log write is the gateway's one privileged act, and it deserves its own probe and a static check that the privileged client is never imported anywhere a read occurs.

03 · Request and Response

Single-turn by design. The request carries an optional conversation_id used only to thread log rows; the gateway holds no memory between calls, and prior turns are not replayed in v1. That exclusion is named rather than implied: continuity is a Phase 02 question with retention implications, not a default.

// request · POST /functions/v1/nou-gateway · [prop]
{
  "question": "When must notice of the annual meeting go out?",
  "context": {
    "surface": "portal",            // portal | hub
    "record": {                      // optional: ground on one record
      "table": "notices",
      "id": "<uuid>"
    }
  },
  "conversation_id": null            // log threading only
}
// response · [prop] · shapes the W1.7 probe asserts
{
  "answer": "Notice of the annual meeting must go out within the
             window set by the bylaws [1], and your own notice
             record shows delivery confirmed [2].",
  "citations": [
    { "n": 1, "kind": "section", "ref": "draft-bylaws:\u00a72.4",
      "label": "Notice timing" },
    { "n": 2, "kind": "record",  "table": "notices", "id": "<uuid>",
      "label": "Your meeting notice, delivered" }
  ],
  "grounding": "grounded",           // grounded | partial | refused
  "refusal": null,                   // or { code, message } per \u00a706
  "interaction_id": "<uuid>",
  "model": "<model string from env>"
}

Citation kinds are two: section citations use the D7 register addresses verbatim, prefix and all, until R1 freezes them; record citations carry table and id plus a human label, and the portal renders them as links into the record detail pattern. The model string is configuration, not code, read from function env so model changes are operational acts that appear in the log, never silent edits prop.

04 · Citation as Obligation

The obligation is enforced after generation, not requested before it. The model is instructed to support every factual claim with a numbered citation, but instructions are weather; the verification pass is climate. Stage 4 takes each emitted citation and resolves it: section citations must exist in the D7 register, and record citations must return a row through the user-scoped client. A citation that fails resolution is stripped. If stripping leaves the answer with claims and no support, the gateway downgrades: grounding falls to partial when some claims survive supported, and the whole response converts to a refusal with code ungrounded when none do prop.

Resolution through the user client carries a subtlety worth naming as finding F11: row-level security returns zero rows both for records that do not exist and for records the caller may not read, and the gateway cannot tell the difference. This is a feature, and the refusal wording in §06 is written around it: Nou never asserts that a record does not exist, only that nothing readable to you supports an answer. Any wording that distinguishes the two cases would leak existence through the side door, exactly what the pairwise probe exists to forbid.

05 · Scoped Impersonation

The property is structural: there is no retrieval path that does not pass through a client carrying the caller's own JWT, so the database treats Nou's reads as the member's reads, and the C7 matrix column for Nou is uniform by construction. The probes restate it three ways (plan W1.7, C7): the pairwise probe runs the full persona suite once directly and once through the gateway and diffs the results, which must be identical; the cross-member probe asks Nou, as member-a, a question answerable only from member-b's rows, and the refusal must match the direct denial; and the privilege probe asserts by static check and by test that the privileged client is reachable only inside log_interaction().

One boundary case is decided here rather than discovered later: when context.record names a row the caller cannot read, stage 2 simply retrieves nothing for it, and the answer proceeds or refuses on what remains. The gateway does not error on the unreadable reference, because a distinct error would be an existence oracle, the same leak F11 guards in stage 4 prop.

06 · Refusal Protocol

Refusal is a success state, not an error: the harness working as designed. Codes are enumerated so the log can be read statistically, and the wording is canonical so refusals cannot drift into either apology or leakage. The member-facing message is always the canonical text plus, where useful, a pointer to a person or a section, never an explanation of what the gateway almost said.

codewhencanonical wording
ungroundedno citation survives verification"Nothing readable to you supports an answer to this. A person who can help: the Secretary."
out_of_scopequestion is not about the cooperative's records or governance"This sits outside what Nou answers. Nou reads the cooperative's own records and documents, nothing wider."
write_requestedthe question asks Nou to do, change, file, or send anything"Nou v1 reads; it does not act. The action you describe belongs to a person, through the portal."
rate_limitedper-user limit reached"Nou has answered a lot for you this hour. Ask again shortly."
upstream_errormodel API failure after retry"Nou could not complete this. The failure is logged; nothing about your question was lost."

All wording prop, offered for adoption with the rest. The rule that generated it: a refusal names what Nou is, points somewhere real, and confirms nothing about rows the caller cannot read. Refusals are logged with their code and grounding: "refused", so the monthly operations review can see what members needed and did not get, which is itself a legibility instrument.

07 · The Log

The committed trio is prompt, citation set, response (plan W1.7); the rest of the shape conforms to the CONVENTIONS event envelope where the envelope is meaningful, and is tagged prop. Retention is the interim posture named in the plan: retain all, administrator-read only, explicitly interim per CIS §17, with the member-reads-own cell standing as proposed in C7 until the retention decision is actually taken.

-- migration · W1.7 · [plan trio + prop shape]

08 · The System Prompt

The template is part of the spec so that changing it is a reviewed act. It is short on purpose: every property that matters is enforced after generation, and the prompt's job is only to make compliance the path of least resistance. Draft v1, prop:

You are Nou, the read-only records instrument of RegenHub, LCA.
Answer the member's question using ONLY the context blocks below.
Every factual claim must carry a numbered citation to a context
block. If the context does not support an answer, say exactly:
UNGROUNDED. Do not use outside knowledge. Do not speculate about
records you were not given. Do not offer to take actions. Plain
prose, no headings, cite sections by their given addresses.

[context blocks: governing-document sections, then caller-own
 records, each with its address or table/id label]

Retrieval composition for v1 is deliberately plain prop: a section index over the agreements repository matched on the question's terms, plus the single record named in context.record when present and readable. No embeddings, no vector store; pgvector remains available in the substrate if relevance quality demands it later, and adopting it would be a milestone, not a drift.

09 · What v1 Is Not, and What Remains Open

Nou v1 does not write, send, file, draft into workflows, remember across calls, browse, or use tools. The drafting duties named elsewhere in the plan, minutes at W4.4 and the impact report at W2.6, are separate harnesses with human ratification at their center, and they will get their own spec lines at those steps rather than inheriting this one by analogy.

Open at W1.7's PR, none rising to a D-level decision: the per-user rate limit number; whether responses stream; the exact section-index method; and the retention decision that would activate the member-reads-own log cell. Two findings continue the series: F10, the log write is the gateway's single privileged act and carries its own probe and static check; F11, RLS cannot distinguish absent from denied, so refusal wording must never claim non-existence, and the §06 texts are written to that rule.
