The Builder's Collection

Artifacts that let agents build the CIS, mapped and sequenced

Version 0.1 · Draft for review · June 2026
01 · What exists

Four artifacts before this turn.

The implementation plan v0.1 (markdown) is the canonical assessment and serialized ledger; it is the source every other artifact derives from. The design-system rendering of the plan (HTML) is its reading copy for humans. The roadmap instrument v0.2 (HTML) is the interactive index, tree, and gate tracker, with plain and technical lenses. And upstream of all three sit the governing sources themselves: CIS PRD v0.2, Hub Operations PRD v0.1, and the canonical co-op.us design-system file.

These are strong on description and weak on execution. An agent handed all four still cannot take W0.1 without inventing the very things that should be contracts: how to behave, what the milestone files literally contain, what each CI job actually asserts, and what the humans have or have not decided.

02 · The gap, stated plainly

Plans describe; agents execute from contracts.

Four layers were missing:

  • Conduct layerAn in-repo operating contract that makes the plan's §08 loop binding, names the hard rules (secrets, RLS, vocabulary, human-only decisions, no weakened verification), and makes refusal a defined good outcome.
  • Executable ledgerThe 67 entries as actual schema-valid files with a working validator, not a document an agent must transcribe. Transcription is where drift enters.
  • Verification layerContract cards for the named CI jobs, without which every success_state is circular.
  • Decision surfaceDeliberation briefs the humans can decide quickly and a record format agents can consume mechanically, since the whole tree currently waits on D3.

A fifth gap is real but deliberately deferred: domain reference (consolidated table schemas, probe matrices, Nou gateway spec, KPI fixtures). Deferred because producing column-level detail now would outrun the PRDs and pre-empt counsel; it lands in the series at the step that consumes it.

04 · What ships in this turn, with proof

The repo seed: 112 files, validator tested.

Verification record

The repo seed (cis-repo-seed-v0.1.zip, 112 files) contains C1, C2, C3, C5, and the conventions half of C4. The validator was exercised here, not merely written:

Green on the full seed — 67 entries, 142 expanded edges, graph acyclic, gating consistent, index.json and STATUS.md regenerated.

A deliberate negative test flipping W1.2 to done failed with the exact gating message (status done but unverified parents: W1.1) and exited nonzero. Restoration returned green.

The validator also caught and survived one of its own bugs during testing, which is recorded honestly: it initially re-ingested its generated STATUS.md, and now excludes generated files by name.

05 · Honest limits

What this seed is, and is not.

  • The Actions workflow is written to the documented GitHub interface but has not run on GitHub from here; its main-branch auto-commit step needs either a bot allowance under branch protection or rerouting through a PR, and the file says so.
  • The milestone schema allows bylaw_refs and prd_refs fields, but citations currently live in brief text; field-level population is hardening work inside W0.2.
  • Decision records are all open, as they must be.
  • Seeded entries (W3.1 to W3.7, W4.1 to W4.7) remain summary-level by design.
  • The seed has not touched a real Supabase project or repository; it is scaffolding, verified as scaffolding.
06 · Frame note

What this series does not cover.

The C-series is ordered by what unblocks agents soonest; a counsel-first or grant-first ordering would sequence differently and is defensible. The series deliberately leaves out program-side systems (Coordination Games), counsel work product, and the design system itself, which already exists as the canonical file and enters the build at W0.13 rather than through this collection.
