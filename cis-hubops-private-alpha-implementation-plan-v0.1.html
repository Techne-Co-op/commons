01 · Frame and Exclusions

The organizing frame below is mine, and it is replaceable. I have kept the PRDs' own phase structure (CIS Phases 01–06, Hub Phases 01–05) as the module grouping, because the PRDs are the authoritative specification. I have re-sequenced delivery into waves, which are dependency strata rather than calendar periods, per the no-time-estimates constraint. Three ledger species sit alongside the work steps: decisions (D-series, human-owned), ratification gates (R-series, counsel-involved), and release gates (G-series, composite). The alternate four-group lens (Membership/Identity, Capital/Value Flow, Governance/Decision, Records/Public Surface) maps onto waves W1/W3, W3, W4, and W6 respectively; I did not use it as the index because the PRD phases carry the CROPS stage targets that the roadmap is evaluated against (CIS §16).

What this plan deliberately excludes, so the omission is visible rather than silent: counsel work product (bylaws ratification, Hub Membership Agreement review) is indexed as gates but not planned; Coordination Games infrastructure is a separate system and out of scope; the Xero general-ledger mirror, ENS resolvability, and on-chain attestation are deferred beyond alpha and indexed; CIS Phase 06 (Practice) is cultural work that cannot be assigned to a software agent and is indexed as a human milestone; and the PRD's Make.com orchestration layer is demoted to an augmentation candidate, which is the single largest departure from the written stack and is argued in 2.2.

02 · Assessment: the PRDs Against the Constrained Substrate

2.1What the PRDs require, in essence

The CIS is the book of record for the cooperative as an entity: a single authoritative Supabase state with three faces (ledger, member portal, Nou), an REA information model of 19 core tables, 13 cooperative modules across five build phases plus a cultural sixth, 11 internal roles enforced at the row level under draft-bylaws §18.1, and the CROPS pathway as the design discipline each phase is graded against (CIS §03–§08, §16). Hub Operations is the book of record for the cooperative as a host: 12 extension tables, 8 modules, 4 additional roles, three computed KPIs, a monthly impact report drafted by Nou and approved by a human, and a grant arc whose second tranche is gated on KPI compliance (Hub §02, §04–§08). The two surfaces share agents, agreements, treasury, and CROPS posture; Hub Operations is greenfield and is meant to set Stage 02–03 patterns that the CIS inherits (CIS §09, Hub §09).

The combined schema surface is 31 tables, all tracing to the four REA primitives, with derived views as pure functions of the event log. Conservation, traceability, and audit live in the database, not the interface (CIS principles 04, 05, 09). That property is what makes the constrained substrate viable at all, as 2.3 explains.

The Make.com demotion deserves its reasoning stated plainly. The PRD's own CR pathway treats every vendor as a chokepoint to be analyzed and names the walkaway test as the Phase 05 criterion (CIS §04). Make.com appears in the PRD as orchestration only: it holds no state and makes no decisions. Everything assigned to it (notice sends, reminders, scheduled jobs, cross-system glue) is expressible in pg_cron, Edge Functions, and Actions workflows, which are already in the principal substrate. Removing it removes a Stage 01 chokepoint at the cost of more first-party code, and the cost is the right kind: code in a public repository advances the O pathway, while a Make.com scenario library does not export cleanly. The adoption trigger that would reverse this is defined under D5 and recorded in the chokepoint analysis, so the demotion is a documented decision, not folklore.

2.3What GitHub Pages actually imposes

Pages is static hosting: no server runtime, no request-time secrets, no custom response headers, no server-side redirects. Four consequences follow, and the first is the one that shapes everything.

First, row-level security is the entire authorization perimeter. The Supabase anon key ships in the public bundle by design; it is safe exactly to the extent that every table is default-deny with explicit, correct policies. This is not a weakening of the PRD; it is the PRD's principle 06 ("privacy is a property of the record, not the screen") taken at its word, with no screen left to hide behind. It does, however, convert RLS completeness from a quality goal into a hard release gate, so the plan makes the automated RLS audit a required CI check that no wave can merge past (W0.6).

Second, the application shell is public. Pages on the current plan publishes publicly; a truly private static shell requires either an enterprise plan or a proxy in front, both of which are augmentations. The honest reading: in this alpha, "private" means private data (auth-gated, invite-only, RLS-enforced) rather than a secret artifact. The CIS open-source pathway already commits the schema and eventually the front-end code to publication (O Phase 01 and Phase 03), so a public shell is on-pathway rather than a concession. The residual exposure is that outsiders can see that an alpha exists and what its screens are shaped like. Whether that is acceptable is D6, with the proxy augmentation named as the alternative.

Third, server-side work moves to two rails. Anything needing secrets or privileged writes (Nou's Claude key, email provider key, Mercury token, service-role operations) lives in Edge Functions; anything needing schedules lives in pg_cron with GitHub Actions cron as the independent fallback rail. Two rails is deliberate: the Reliability requirement (CIS §14) asks that no single vendor failure invalidate state, and a scheduled export that runs from Actions still fires if the database's own scheduler is the thing that failed.

Fourth, small mechanics: SPA routing needs a 404.html fallback or hash routing; build-time environment injection replaces runtime configuration; a meta-tag CSP substitutes for header CSP; the alpha site carries noindex. Each appears as an explicit success condition in W0.12 rather than as tribal knowledge.

2.4What "production in private alpha" means here

One Supabase project designated production-of-record from the first migration (local development through the Supabase CLI; no second hosted environment in alpha, which is named as an accepted risk in 2.5). One public repository, one Pages site on a cooperative subdomain. Real members under invite-only auth; synthetic data only until the moment of first real onboarding, which is itself a gated step. The alpha gate G1 (section 07) defines the verified step set that constitutes "deployed."

03 · Version Control as the Milestone System

The repository is the milestone system. There is no separate tracker: every step in section 06 is a file, every status change is a commit, and the audit trail of the build is the git history itself. This satisfies "organize, serialize, and index in a version control system" by construction, and it makes the build auditable with the same gesture that makes it trackable.

3.2Milestone file schema

Every ledger entry is a markdown file with YAML front matter. The front matter is the machine surface; the body is the agent brief.

---
id: W1.02                # unique, immutable
title: memberships table and class model
wave: W1
owner: agent             # agent | human | counsel
status: pending          # pending | in_progress | blocked | review | done | verified
depends_on: [W0.07, W0.06]
decision_refs: []        # D-items this step consumes
bylaw_refs: [draft-bylaws:1.1-1.4, draft-bylaws:18.1]
prd_refs: [CIS §06, CIS §08 Member Directory]
success_state: >
  One-sentence, testable condition.
verification: ci:db-verify   # ci:<job> | human:<role> | counsel
artifacts: [supabase/migrations/*_memberships.sql]
---
Context. Why this exists, what it touches, what it must not touch.
Action. What the executing agent does, concretely.
Notes. Simplifications taken, tensions left open.

Body sections are the contextual information each directed agent needs; for waves W0–W2 this document seeds them in full (section 06), and for W3–W6 the first step of each wave is the authoring of the remaining briefs, so detail is produced at the boundary where it is real rather than invented here.

3.3Status semantics and gating

pending → in_progress → review → done → verified, with blocked available from any state when a decision_ref or dependency is unmet (the blocking reason is recorded in the file). Two rules carry the audit property. A step may not leave pending until every entry in depends_on is verified. And done (work merged) is distinct from verified (success state demonstrated): verified requires either the named CI job green on the merged commit, or for human:/counsel verification, an approving PR review by the named role recorded on the status-flip commit. Only verified unlocks dependents. Status changes happen only by editing the milestone file in the same PR as the work, so the graph state and the code state can never diverge.

3.4The validator and the near-real-time board

A milestone-validate workflow runs on every push and PR: it parses all front matter, enforces the schema, asserts the dependency graph is acyclic, asserts the gating rules above, and regenerates index.json and STATUS.md (committed by the workflow on main). A status-mirror job then POSTs index.json to a Supabase build_status table via Edge Function. The portal's /build panel renders that table over Realtime, which closes the loop the legibility thesis asks for: the cooperative's information system displays the live state of its own construction, to the same members it is being built for. Near real time, in practice, means every merge.

3.5Commit, branch, and PR conventions

Branches: step/W1.02-memberships. Commits: type(W1.02): subject with conventional types. Every PR template requires the step id, the success-state evidence (CI link or reviewer), and the status flip in the milestone file. Branch protection on main requires milestone-validate, db-verify, rls-audit, grammar-lint, and portal-build green. One step per PR as the norm; a PR touching code without touching a milestone file fails validation, which keeps the ledger exhaustive.

05 · Decision and Ratification Ledger

Every item here is human-owned. Defaults are proposals only; I have flagged where I extended the PRD's option set so the extension is a visible choice rather than a smuggled one.

D1 · front-end framework

Owner: organizers. The PRD leaves the tool open with criteria: portability, accessibility, mobile-first, Supabase compatibility, design-system fidelity, plus the chokepoint test (CIS §17). Proposed default: a static-first framework with islands of interactivity (Astro is the type specimen; a Vite + React SPA is the alternative). The criteria favor static-first because Pages serves HTML, the design system is typographic and editorial, and less client JS is less surface. Blocks W0.12 onward.

D2 · application form host

Owner: organizers. The PRDs frame this as Formbricks (self-host) versus Tally/Typeform (managed chokepoint) (Hub §11). The constraint adds a third option not in the PRD, and I am naming that addition explicitly: a static form on Pages inserting through a narrowly scoped anonymous RLS policy into hub_applications. No new vendor, open-source by default, advances CR and O simultaneously. Proposed default: the native path, with Formbricks as the fallback if form complexity outgrows it. Blocks W2.2 and CIS admission (W1.6) inherits the pattern (CIS §17).

D3 · repository visibility and naming

Owner: organizers. Proposed default: public from first commit, which is the O-pathway Phase 01 commitment ("the schema itself is published," CIS §04) made literal. Hard rule either way: no secrets, no member PII, synthetic seeds only. Blocks W0.1.

D4 · transactional email provider

Owner: organizers. One provider with API send and delivery webhooks (Resend, Postmark, and SES are the candidate class; no endorsement implied). Required because notices carry legally timed windows and delivery tracking (CIS §08, draft-bylaws §2.4) and Supabase's built-in mail covers auth flows only. Blocks W0.11, therefore W1.8.

D5 · make.com adoption trigger

Owner: organizers. Proposed default: not adopted in alpha. Reversal trigger, to be recorded in docs/chokepoint-analysis.md if invoked: a required integration whose first-party Edge Function cost exceeds reasonable effort and whose data remains exportable. Until invoked, the PRD's Make.com responsibilities route to pg_cron, Edge Functions, and Actions.

D6 · alpha access posture

Owner: organizers. Proposed default: accept the public static shell; privacy enforced at data layer; noindex; invite-only auth allowlist. Alternative: a proxy with access control in front of Pages, named as an augmentation. Blocks the public exposure of W0.12.

D7 · section addressing scheme

Owner: organizers with counsel awareness. Pre-ratification, every bylaw citation in policies, schema comments, and UI uses the draft-bylaws:§x.y prefix against a register in docs/citations.md; ratification (R1) triggers a single sweep that drops the prefix and freezes addresses. This keeps cite-as-you-enforce honest before the text is final. Blocks W0.6's citation convention.

D8 · attestation scope and key custody

Owner: working group per CIS §17. Deferred; blocks only the optional attestation extensions in W4 and W6.

D9 · EE liaison interface form

Owner: organizers with EE team. Dashboard view versus digest versus shared page; CROPS implications cut both ways (Hub §11). Proposed default: scoped read-only dashboard credential, with the public KPI view satisfying the open-source pull. Blocks W2.9's liaison surface.

D10 · lease-entity question

Owner: board. Whether West or the LCA holds the third-floor lease affects how facility obligations enter the ledger (CIS §17). Blocks specific W3 treasury detail steps only; indexed so the dependency is visible.

R1 · bylaws ratification

Owner: counsel and membership (Pote Law Firm, Yev Muchnik named reviewers). Activates: allocation-formula automation (until then, manual allocation entry against the equal default, exactly as CIS §17 specifies), citation freeze (D7 sweep), and final §18.1 role semantics. Does not block any build step.

R2 · hub membership agreement

Owner: counsel (Pote review per Hub §11). Until ratified, Hub access proceeds against the interim acceptable-use policy, recorded as an agreement-repository entry so the interim state is itself on the record.

G1 · alpha gate

Composite; defined in section 07.

P1 · practice phase

Owner: membership. CIS Phase 06 is deliberately not a development phase (CIS §16); indexed so the ledger does not pretend the work ends when the architecture does.

07 · G1: the Alpha Gate

G1 is verified when every step in the following set is verified: all of W0; W1.1 through W1.8; W2.1 through W2.6; and decisions D1, D2, D3, D4, D6, D7 recorded. W2.7 through W2.9 and all of W3 onward are explicitly not required for G1; they proceed against the same ledger after the gate. The gate's own milestone file lists the set, so the validator computes G1 status mechanically and the /build panel shows distance-to-alpha as a single number. Operationally, G1 means: a real invited member can sign in, see what governs them, sign their agreement, appear in the directory, ask Nou a grounded question and get a cited answer, and the Hub team can run applications, sessions, events, the KPI dashboard, and a monthly report against production data.

08 · Operating the Ledger

For an executing agent, the loop is: take the lowest-indexed pending step whose dependencies are verified and whose decision_refs are resolved; branch as step/<id>; do the Action; flip the status in the same PR; merge only when the named verification is green; the validator advances the board. For Todd, the loop is: STATUS.md or the /build panel for state, the D and R ledger for what is waiting on a human, and PR review wherever verification: human names a role. Anything an agent cannot complete becomes blocked with its reason written into the file rather than worked around, which keeps the dependency tree truthful. The plan itself is versioned: amendments to this document are commits like any other, so the history of the plan is part of the audit trail of the build.
