## What this step does

Introduces `build_site.py` — plain Python (PyYAML + stdlib string.Template
or Jinja2) — that renders the site's generated regions from `milestones/`
and `docs/decisions/` at deploy time. Introduces `pages-deploy.yml`, an
Actions-based GitHub Pages workflow that runs after `milestone-validate`
passes on main.

## Why

The public surface currently holds three copies of the ledger's truth: the
canonical front matter, the generated `index.json`/`STATUS.md`, and
hand-painted snapshots in the HTML files. The third copy has already diverged.
The site becomes a derived view: authored prose is hand-edited; status counts,
decision rows, wave cards, and the repo-tree diagram are generated.

## What is generated vs authored

Generated: live status counts, decision register rows, wave grid counts,
repo-tree diagram, document library list (from manifest.yaml), roadmap
instrument dataset (const L = injected from index.json).

Authored (hand-edited prose, preserved): the REA explanation, the "why now"
narrative, the co-op orientation text.

## Constraint: D1 remains open

This pipeline must not smuggle a framework decision in through the
documentation site. `build_site.py` is plain Python. The member portal
(W0.12) remains free to choose its own tooling under D1; whether it later
shares this Pages property or stands separately is an organizer call.

## Deployment model

Moves from branch-root serving to Actions-based Pages
(`upload-pages-artifact` → `deploy-pages`). Generated site files are never
committed back to main. This decouples the public URL layout from the
repository layout and dissolves the branch-protection caveat in the existing
workflow's comments.
